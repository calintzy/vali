# Vali — AI 생성 코드 검증 자동화 도구

## 1. 프로젝트 개요

| 항목 | 내용 |
|------|------|
| **이름** | Vali (Validate AI의 약자) |
| **한줄 소개** | AI가 생성한 코드의 환각, 슬롭, 과잉설계를 자동 감지하는 특화 린터 |
| **카테고리** | Developer Tooling / AI Code Quality |
| **라이선스** | MIT |
| **난이도** | 중 |
| **예상 개발 기간** | MVP 4~6주 |

## 2. 문제 정의

### 현재 상황 (2026년)
- 기업 코드의 **40%+**가 AI 생성 (GitHub Copilot, Claude, Cursor 등)
- AI 생성 코드의 **38%**가 "인간 코드보다 리뷰가 더 힘들다"고 응답 (Stack Overflow 2025 Survey)
- AI 코드에는 기존 린터가 잡지 못하는 **고유한 패턴**이 있음

### AI 코드의 고유 문제 패턴

| 패턴 | 설명 | 기존 린터 감지 |
|------|------|--------------|
| **환각 Import** | 존재하지 않는 패키지/모듈을 import | ❌ (타입체크만 가능) |
| **빈 함수/스텁** | 구현 없이 TODO만 있는 함수 | ❌ |
| **과잉 주석** | 코드보다 주석이 더 많은 AI 특유 패턴 | ❌ |
| **과잉 설계** | 단순한 기능에 불필요한 추상화/패턴 | ❌ |
| **유사 반복** | 거의 같은 코드를 약간씩 변형 반복 | △ (일부 감지) |
| **죽은 파라미터** | 사용되지 않는 함수 인자 | △ |
| **환각 API** | 실제로 없는 메서드/프로퍼티 호출 | ❌ (타입체크만) |
| **과잉 에러 핸들링** | 불필요한 try-catch, 불가능한 에러 케이스 처리 | ❌ |

### 핵심 고통점
1. **리뷰 부담 증가**: AI가 코드를 만들지만, 검증은 사람이 해야 함
2. **기존 도구 부적합**: ESLint는 문법, TypeScript는 타입만 — AI 패턴 전용 도구 없음
3. **팀 표준 위반**: AI가 프로젝트 컨벤션을 무시하고 자기만의 스타일 생성
4. **코드 비대화**: AI가 생성한 불필요한 코드가 쌓여 기술 부채 증가

## 3. 솔루션

### 핵심 컨셉
```bash
$ vali check src/

🔍 Vali: AI Code Quality Report
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

src/api/handler.ts
  ⛔ L3: Hallucinated import — 'express-validator/magic' does not exist
  ⚠️ L15-42: Over-engineered — Simple CRUD wrapped in Strategy Pattern + Factory
  ⚠️ L56: Empty function — processData() has only a TODO comment
  💬 L8-14: Comment bloat — 7 lines of comments for 3 lines of code

src/utils/helpers.ts
  ⛔ L22: Hallucinated API — Array.prototype.groupBy is not standard
  ⚠️ L30-55: Near-duplicate — 92% similar to src/utils/formatters.ts:10-35
  💬 L1-5: AI signature — "This file contains utility functions for..." boilerplate

src/db/queries.ts
  ⚠️ L45: Dead parameter — 'options' argument is never used
  ⚠️ L67-89: Excessive error handling — catches impossible error cases

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Summary: 3 files | 2 errors | 5 warnings | 2 info
AI Slop Score: 34/100 (moderate — needs cleanup)
```

### AI 슬롭 감지 규칙

| 규칙 ID | 이름 | 심각도 | 설명 |
|---------|------|-------|------|
| `VAL001` | hallucinated-import | error | 존재하지 않는 패키지/모듈 import |
| `VAL002` | hallucinated-api | error | 존재하지 않는 메서드/프로퍼티 사용 |
| `VAL003` | empty-function | warning | 구현 없는 함수 (TODO만 있음) |
| `VAL004` | comment-bloat | info | 코드 대비 과도한 주석 비율 |
| `VAL005` | over-engineered | warning | 불필요한 디자인 패턴/추상화 |
| `VAL006` | near-duplicate | warning | 높은 유사도의 코드 블록 |
| `VAL007` | dead-parameter | warning | 사용되지 않는 함수 인자 |
| `VAL008` | excessive-error | warning | 불필요한 에러 핸들링 |
| `VAL009` | ai-boilerplate | info | "This file contains..." 등 AI 특유 서문 |
| `VAL010` | unused-abstraction | warning | 한 번만 사용되는 추상화/인터페이스 |

### 작동 방식
```bash
vali check src/                    # 디렉토리 스캔
vali check --fix src/              # 자동 수정 (안전한 것만)
vali check --diff                  # git diff 기준 (변경된 파일만)
vali check --ci                    # CI 모드 (JSON 출력, exit code)
vali init                          # 프로젝트별 설정 파일 생성
vali rules                         # 활성 규칙 목록
```

## 6. 기술 스택

| 구성 요소 | 기술 | 선택 이유 |

### Vali의 차별점
1. **AI 전용**: 기존 린터가 못 잡는 AI 코드 고유 패턴에 집중
2. **다국어**: TypeScript, Python, (향후) Go, Rust 지원 목표
3. **Actionable**: 감지 + 자동 수정 + 리팩토링 제안
4. **2026 시대 적합**: AI 코드 비율 급증 → 수요 폭발 예상

## 8. 구현 계획

### Phase 1: 환각 감지 (1~2주)
- CLI 프레임워크 구축
- TypeScript AST 파서 통합
- VAL001 환각 import 감지 (node_modules 존재 확인)
- VAL003 빈 함수 감지 (AST에서 빈 body 탐지)
- VAL004 주석 비율 분석

### Phase 2: 패턴 감지 (2~3주)
- VAL005 과잉 설계 감지 (Strategy, Factory 등 불필요 패턴)
- VAL006 유사 코드 블록 감지
- VAL009 AI 보일러플레이트 감지
- `.valirc` 설정 파일 시스템

### Phase 3: 자동화 (3~5주)
- `--fix` 자동 수정 (안전한 규칙만)
- `--diff` Git 변경분만 체크
- JSON/SARIF 출력 (CI 통합)
- ESLint 플러그인 형태 제공

### Phase 4: 출시 (5~6주)
- npm 배포 (`npx vali check src/`)
- GitHub Action 제작
- 유명 AI 생성 코드로 데모 제작
- README + 문서

## 10. 리스크 및 대응

| 리스크 | 확률 | 대응 |
|--------|-----|------|
| "AI인지 사람인지" 구분 논란 | 높음 | AI 여부 판단이 아닌, "코드 품질 패턴" 감지로 포지셔닝 |
| ESLint/Biome가 AI 규칙 도입 | 중간 | 선점 + 전용 도구의 깊이로 차별화 |
| false positive 높을 수 있음 | 높음 | 보수적 기본값 + 사용자 피드백으로 규칙 조정 |
| 빠르게 변하는 AI 코딩 패턴 | 중간 | 커뮤니티 기반 규칙 추가 시스템 |
